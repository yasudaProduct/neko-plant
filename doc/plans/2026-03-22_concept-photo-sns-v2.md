# リデザイン構想 v2: 植物×猫 フォトSNS コンセプトドキュメント

作成日: 2026-03-22

---

## 1. コアコンセプト

### 「猫と植物の暮らしを共有する」

現行の neko-plant は「植物の安全性データベース」として機能しているが、v2 では **フォトSNS** へと転換する。

- ユーザーは猫と植物が一緒にいる写真を投稿する（Instagram的）
- 投稿にはAI or 手動で植物名がタグ付けされる
- 「この植物と猫が一緒に暮らしている」= **暗黙の安全性エビデンス**
- 投稿が集まるほど、植物ごとの安全性が自然にわかる

### パラダイムシフト

| 項目 | 現行（v1） | v2 |
|------|-----------|-----|
| 主な行動 | 植物を検索 → 評価を読む | フィードを眺める → 投稿する |
| 安全性の表現 | good/bad の明示的評価 | 共存実績の多さ（ポジティブリスト） |
| 投稿の起点 | 植物詳細ページから評価 | 写真を撮ってそのまま投稿 |
| コンテンツの主役 | 植物データ | 猫と植物の写真 |
| ユーザー体験 | データベース検索ツール | SNSフィード |

---

## 2. ターゲットユーザーと動機

| ユーザー層 | 動機 | 主な行動 |
|-----------|------|---------|
| 猫×植物の飼い主 | かわいい写真を共有したい | 投稿する |
| 癒されたい人 | 猫と植物の写真を見たい | フィードを眺める |
| 安全性を調べたい人 | 新しい植物を買いたい | 植物名で検索する |

**ポイント:** 3つ目のユーザー層は現行v1のコアユーザーと同じ。v2ではこの層に加えて、1つ目・2つ目の層を取り込む。安全性情報は「投稿の副産物」として自然に蓄積される。

---

## 3. 安全性の考え方

### ポジティブリスト方式

従来の「good/bad評価」から、**共存実績の多さ**による安全性表現へ転換する。

```
投稿が多い植物 → 多くの猫と共存実績あり → おそらく安全
投稿がない植物 → 誰も一緒に飼っていない → 不明 or 危険の可能性
```

### ユーザーへの見せ方

- 「モンステラ: 234匹の猫と暮らしています 🐱🌿」
- 「ユリ: 投稿がありません ⚠️ 猫との共存情報がないため注意してください」

### 設計上の原則

1. **明示的に「危険」と断言しない** — 「情報がない」と表現する（法的リスクの回避）
2. **外部データとの連携余地** — ASPCA等の公式データベースで毒性が確認されている植物には、外部情報として警告を出せると良い（将来的な拡張ポイント）
3. **共存実績 = 投稿数 × ユニーク猫数** — 同じユーザーの重複投稿ではなく、異なる猫との実績を重視

## 4. 主要機能

### 4.1 フィード（ホーム）

SNSの中心となるタイムライン。ユーザーが最初に目にする画面。

- **新着投稿フィード** — 最新の猫×植物の写真が並ぶ
- **写真中心のレイアウト** — Instagram的なカード型 or グリッド型
- **各投稿の表示要素:**
  - 写真（メイン）
  - 投稿者情報（アイコン、名前）
  - 植物名タグ
  - 猫の名前・種類
  - いいね数
  - 投稿日時

### 4.2 投稿

写真起点のシンプルな投稿フロー。

- **フロー:** 写真選択 → AI植物判定 or 手動タグ付け → 猫を選択 → コメント（任意）→ 投稿
- **既存の写真起点投稿（`/posts/new`）をベースに拡張**
- v1の「good/bad評価」は廃止 — 投稿すること自体が「この植物と猫は共存している」という暗黙の安全性エビデンスになる

### 4.3 植物ページ

植物ごとの情報が集約されるページ。v1の植物詳細ページの進化形。

- **共存実績サマリー:** 「XX匹の猫と暮らしています」
- **関連投稿一覧:** この植物がタグ付けされた投稿のギャラリー
- **植物情報:** 学名、科、属、種（既存データを活用）
- **安全性注記:** 投稿がない場合の注意喚起

### 4.4 ユーザープロフィール

- **投稿一覧:** そのユーザーの投稿ギャラリー
- **猫プロフィール:** 飼っている猫の情報（既存のpets機能を活用）
- **植物コレクション:** 一緒に暮らしている植物の一覧（投稿から自動集計）

### 4.5 検索・探索

- **植物名検索** — v1の検索機能をベースに、共存実績数でソート
- **タグ検索** — 植物タグによる投稿の絞り込み
- **猫種検索** — 特定の猫種 × 植物の組み合わせ

## 5. 現行機能との対応関係

### 廃止する機能

| 現行機能 | 理由 |
|---------|------|
| good/bad 評価投稿 | ポジティブリスト方式に転換。投稿 = 共存エビデンスになる |
| 評価リアクション（good/bad） | いいね（単一種類）に置き換え |
| plant_have（植物所持） | 投稿から自動的に判別できるため不要 |

### 維持・進化する機能

| 現行機能 | v2での扱い |
|---------|-----------|
| 植物カタログ（plants） | 維持。植物ページとして進化 |
| 植物検索 | 維持。共存実績数ベースのソートを追加 |
| AI植物判定 | 維持。投稿フローに組み込み済み |
| 猫プロフィール（pets） | 維持。投稿と猫を紐付ける基盤として活用 |
| 猫種マスタ（neko） | 維持 |
| ユーザープロフィール | 維持。投稿ギャラリーとして進化 |
| お気に入り（plant_favorites） | 投稿へのいいねに進化 |
| 画像アップロード | 維持。投稿のメイン機能として活用 |
| 管理者機能 | 維持。投稿モデレーションに適応 |

## 6. データモデルへの影響

### 6.1 新規テーブル

#### `posts` — 投稿（コアテーブル）

現行の `evaluations` を置き換える、v2のコアテーブル。

| カラム | 型 | 説明 |
|-------|-----|------|
| id | Int (PK) | 投稿ID |
| user_id | Int (FK → users) | 投稿者 |
| plant_id | Int (FK → plants) | タグ付けされた植物 |
| pet_id | Int? (FK → pets) | 写真に写っている猫（任意） |
| comment | String? | コメント（任意） |
| created_at | DateTime | 投稿日時 |

#### `post_images` — 投稿画像

| カラム | 型 | 説明 |
|-------|-----|------|
| id | Int (PK) | 画像ID |
| post_id | Int (FK → posts) | 紐づく投稿 |
| image_url | String | 画像URL |
| order | Int | 表示順 |
| created_at | DateTime | 作成日時 |

#### `post_likes` — いいね

現行の `evaluation_reactions`（good/bad 2種類）を、単純な「いいね」に置き換え。

| カラム | 型 | 説明 |
|-------|-----|------|
| id | Int (PK) | いいねID |
| post_id | Int (FK → posts) | 対象投稿 |
| user_id | Int (FK → users) | いいねしたユーザー |
| created_at | DateTime | いいね日時 |

### 6.2 既存テーブルの扱い

| テーブル | 判断 | 理由 |
|---------|------|------|
| `plants` | **維持** | 植物マスタとして引き続き利用 |
| `public_users` | **維持** | ユーザー管理の基盤 |
| `neko` | **維持** | 猫種マスタ |
| `pets` | **維持** | 猫プロフィール。投稿との紐付けに活用 |
| `evaluations` | **削除** | `posts` で置き換え |
| `evaluation_reactions` | **削除** | `post_likes` で置き換え |
| `plant_images` | **削除** | `post_images` に統合（投稿画像 = 植物ギャラリー） |
| `plant_favorites` | **削除** | `post_likes` に機能統合 |
| `plant_have` | **削除** | 投稿から自動集計（植物を所持している = その植物でpostがある） |

### 6.3 既存データの移行

| 移行元 | 移行先 | 方針 |
|--------|--------|------|
| `evaluations` | `posts` | type(good/bad)は移行しない。コメントと画像を投稿として移行 |
| `evaluation_reactions` | `post_likes` | good リアクションのみ like として移行 |
| `plant_images`（承認済み） | `post_images` | 承認済み画像を投稿として移行 |

## 7. 安全性指標の算出ロジック

### 共存実績スコア

```
共存猫数 = COUNT(DISTINCT posts.pet_id) WHERE posts.plant_id = ?
共存投稿数 = COUNT(posts.id) WHERE posts.plant_id = ?
```

### 表示ランク

| 共存猫数 | 表示 |
|---------|------|
| 50以上 | 「多くの猫と暮らしています 🐱🌿」（高い共存実績） |
| 10〜49 | 「XX匹の猫と暮らしています」（共存実績あり） |
| 1〜9 | 「XX匹の猫との暮らしが報告されています」（少数の実績） |
| 0 | 「猫との共存情報がありません ⚠️ 注意してください」 |

※ 閾値は運用データを見ながら調整。上記は初期案。

## 8. サービスアーキテクチャ再設計との関係

同日作成の `2026-03-22_redesign-service-architecture.md` では、サービスレイヤー分離の技術設計を記載した。本コンセプトにより、以下の調整が必要:

### 調整が必要な項目

| 設計書の項目 | 調整内容 |
|------------|---------|
| `post-service.ts` | `evaluations` ではなく `posts` テーブルを対象にする |
| データモデル変更 | `plant_have` → `plant_owns` は不要になる（テーブル自体を廃止） |
| `plant_favorites` | 維持ではなく廃止（`post_likes` に統合） |
| `evaluation_reactions` | 維持ではなく廃止（`post_likes` に統合） |
| エラーハンドリング・サービスレイヤーの構造 | そのまま適用可能 |

### サービスレイヤーの最終構成

```
src/services/
├── auth-service.ts       # 認証・認可（変更なし）
├── plant-service.ts      # 植物ドメイン + 共存実績集計
├── post-service.ts       # 投稿CRUD + いいね（posts/post_images/post_likes）
├── user-service.ts       # ユーザードメイン（変更なし）
├── pet-service.ts        # ペット管理（変更なし）
├── storage-service.ts    # ストレージ操作（変更なし）
└── errors.ts             # エラー定義（変更なし）
```

## 9. 実装の優先順位

このブランチで一括実装する際の推奨順序:

1. **データモデル構築** — `posts`, `post_images`, `post_likes` テーブル作成、旧テーブル削除
2. **サービスレイヤー基盤** — errors.ts, auth-service.ts, storage-service.ts
3. **投稿機能** — post-service.ts + post-action.ts + 投稿ページ
4. **フィード** — ホームページをフィード型に変更
5. **植物ページ** — 共存実績表示に変更
6. **ユーザープロフィール** — 投稿ギャラリー化
7. **検索** — 共存実績ベースのソート追加
8. **管理者機能** — 投稿モデレーション対応
9. **データ移行** — 旧evaluations → posts の移行スクリプト
10. **テスト** — サービスレイヤー + E2E
